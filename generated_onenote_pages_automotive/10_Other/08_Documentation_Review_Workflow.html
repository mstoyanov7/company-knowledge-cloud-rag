Documentation Review Workflow

Explains how technical documents are reviewed, approved, and refreshed.

Reference Note

Documentation Review Workflow is a shared reference for teams that need the same wording, decision, or convention. This page explains how technical documents are reviewed, approved, and refreshed.

It should be short enough to stay readable but specific enough that two teams do not interpret the same term or rule differently. The owner should keep examples current because examples are usually what people copy.

How Teams Use It

Teams usually open this page while preparing a review, naming something new, writing a decision record, or checking an audit detail. The useful material is review owner, approval state, refresh date.

When the reference affects repositories, artifacts, or architecture records, update Confluence, GitHub Enterprise, and Jira in the same change window.

Maintained By

Owner teamEngineering Operations
Audienceall technical employees
Review rhythmReview quarterly or when the owner team changes process.
Primary systemsConfluence, GitHub Enterprise, Jira

Useful Material

  • Review owner maintained by Engineering Operations.
  • Approval state available before work starts.
  • Refresh date linked from the ticket, request, or workspace.
  • Current owner, review date, linked decision, and affected teams.

Review Habit

When this reference changes, link the decision or review note in Confluence and tell the teams that depend on GitHub Enterprise. If the change affects artifacts, also update Jira.

Checks

  • Owner team is Engineering Operations and has accepted the request or review.
  • Required artifacts are available: review owner, approval state, refresh date.
  • Relevant systems are checked: Confluence, GitHub Enterprise, Jira.
  • The final decision or completion note is attached to the ticket, request, or record.
