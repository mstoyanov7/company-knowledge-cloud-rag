Test Bench Reservation Tool

Defines how teams reserve HIL rigs, ECU benches, vehicles, and lab equipment for validation work.

Tool Owner Note

Test Bench Reservation Tool is maintained for teams that need the tool during software delivery, validation, or support. This page defines how teams reserve HIL rigs, ECU benches, vehicles, and lab equipment for validation work.

The important distinction is access level. A developer who only reads data should not receive the same role as a release owner, tool maintainer, or audit reviewer.

Request Flow

The request should name the exact target system, usually Bench Scheduler, and include bench ID and reservation slot. If the tool is connected to source code, vehicle data, or supplier evidence, the owner should prefer a group role over one-off manual access.

After access is granted, the requester should open the tool once and confirm that the expected project, namespace, or dashboard is visible. That small check prevents a support ticket from being closed while the user still cannot do the work.

Access Review Points

  • Owner team is Validation Operations and has accepted the request or review.
  • Required artifacts are available: bench ID, reservation slot, test plan.
  • Relevant systems are checked: Bench Scheduler, Lab Inventory, Calendar.
  • The final decision or completion note is attached to the ticket, request, or record.

Operational Habit

When the tool is used during release or validation work, link the ticket back to bench ID and keep the owner visible. That makes later audits easier because the access reason, the system, and the responsible team stay together.

Things That Usually Go Wrong

  • Tool access should follow least privilege and be reviewed when roles change.
  • Shared accounts should be avoided unless explicitly approved.
  • Tool owners are responsible for stale access cleanup.
