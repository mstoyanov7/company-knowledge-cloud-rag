Internal DevTools CLI Setup

Sets up the internal command-line tools used for project bootstrap, diagnostics, and environment checks.

Project Context

Internal DevTools CLI Setup is used when an engineer needs a working local copy of the project, not just read access to the repository. This page sets up the internal command-line tools used for project bootstrap, diagnostics, and environment checks.

The setup normally starts with access to Developer CLI, then moves through the required artifacts: CLI package, profile config, doctor report. The important part is to keep the machine, the project configuration, and the verification output aligned. If those three do not match, the project can appear to work locally and still fail during integration.

Developer Platform keeps this page current because small toolchain changes matter in automotive work. A different compiler, DBC file, ODX package, schema version, or bench profile can change the result enough to waste a full validation session.

Before Opening a Terminal

Confirm that your account is already in the right access group for Developer CLI. Do that before opening a terminal, because clone and package errors often look like network problems when they are actually permission problems.

Create a clean workspace and avoid reusing an old directory from another vehicle program. Keep CLI package and profile config next to the setup ticket so another engineer can reproduce the same run later.

Windows Setup

Use this path on a company Windows laptop or before switching into WSL.

$workspace = Join-Path $env:USERPROFILE "automotive\internal-devtools-cli-setup"
New-Item -ItemType Directory -Force $workspace | Out-Null
Set-Location $workspace
git config --global core.longpaths true
winget install -e --id Python.Python.3.12
py -3.12 -m pip install --upgrade company-devtools
Write-Output "Run the Linux or project-specific verification block after tools are installed."

CLI Install

Use this path on Linux, WSL, or a standard validation workstation.

python -m pip install --upgrade company-devtools
company-devtools profile create automotive-dev --region eu
company-devtools auth login
company-devtools doctor --include secrets,artifacts,git
company-devtools bootstrap --workspace ~/automotive/devtools-check

Verification Commands

Run verification after setup and again after changing profiles, schemas, or hardware mappings.

pytest tests -q
tar -czf internal-devtools-cli-setup-setup-evidence.tgz logs reports

After the First Build

After the first successful build, attach the terminal output and the generated local config to the setup ticket. The expected evidence is setup ticket, access approval, bootstrap output, verification log, plus any project-specific logs created by Artifactory or Secrets Vault.

Do not clean the workspace until the verification step has passed once. Keeping the first build directory for a day makes it easier for Developer Platform to compare cache files, generated code, downloaded packages, or tool versions if something fails for the next engineer.

Checks To Keep With The Ticket

  • Workspace for Internal DevTools CLI Setup builds or starts without missing dependencies.
  • Access to Developer CLI, Artifactory, Secrets Vault is confirmed.
  • Required artifacts are present: CLI package, profile config, doctor report.
  • Setup ticket contains bootstrap output and final verification result.

Typical Setup Problems

  • Outdated bootstrap commands can create inconsistent local environments.
  • Missing profile config usually causes setup to pass locally but fail in integration.
  • Access should be removed when the engineer leaves the project or supplier engagement.

When To Ask For Help

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to Developer Platform. The usual requester is the engineer joining the project, and the standard service desk or program channel should hold the final note.
