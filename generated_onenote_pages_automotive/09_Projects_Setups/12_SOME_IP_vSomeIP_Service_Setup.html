SOME/IP vSomeIP Service Setup

Sets up a SOME/IP service using vSomeIP for service discovery and integration testing.

Project Context

SOME/IP vSomeIP Service Setup is used when an engineer needs a working local copy of the project, not just read access to the repository. This page sets up a SOME/IP service using vSomeIP for service discovery and integration testing.

The setup normally starts with access to vSomeIP, then moves through the required artifacts: vSomeIP repo, service config, network route. The important part is to keep the machine, the project configuration, and the verification output aligned. If those three do not match, the project can appear to work locally and still fail during integration.

Middleware Team keeps this page current because small toolchain changes matter in automotive work. A different compiler, DBC file, ODX package, schema version, or bench profile can change the result enough to waste a full validation session.

Before Opening a Terminal

Confirm that your account is already in the right access group for vSomeIP. Do that before opening a terminal, because clone and package errors often look like network problems when they are actually permission problems.

Create a clean workspace and avoid reusing an old directory from another vehicle program. Keep vSomeIP repo and service config next to the setup ticket so another engineer can reproduce the same run later.

Windows Setup

Use this path on a company Windows laptop or before switching into WSL.

$workspace = Join-Path $env:USERPROFILE "automotive\some-ip-vsomeip-service-setup"
New-Item -ItemType Directory -Force $workspace | Out-Null
Set-Location $workspace
git config --global core.longpaths true
winget install -e --id Git.Git
winget install -e --id Kitware.CMake
winget install -e --id Ninja-build.Ninja
winget install -e --id WiresharkFoundation.Wireshark
git clone ssh://git.company.local/middleware/someip-vsomeip-service.git .
Write-Output "Run the Linux or project-specific verification block after tools are installed."

vSomeIP Service Build

Use this path on Linux, WSL, or a standard validation workstation.

mkdir -p ~/automotive/someip-vsomeip-service
cd ~/automotive/someip-vsomeip-service
git clone ssh://git.company.local/middleware/someip-vsomeip-service.git .
cmake -S third_party/vsomeip -B build/vsomeip -DCMAKE_BUILD_TYPE=Release
cmake --build build/vsomeip --parallel
cmake -S . -B build -DVSOMEIP_ROOT=$PWD/build/vsomeip
cmake --build build --target service_smoke_test --parallel
./build/bin/service_smoke_test --config configs/local.json

Verification Commands

Run verification after setup and again after changing profiles, schemas, or hardware mappings.

./build/bin/service_smoke_test --config configs/local.json
tshark -i any -f "udp port 30490" -a duration:10

After the First Build

After the first successful build, attach the terminal output and the generated local config to the setup ticket. The expected evidence is setup ticket, access approval, bootstrap output, verification log, plus any project-specific logs created by CMake or Wireshark.

Do not clean the workspace until the verification step has passed once. Keeping the first build directory for a day makes it easier for Middleware Team to compare cache files, generated code, downloaded packages, or tool versions if something fails for the next engineer.

Checks To Keep With The Ticket

  • Workspace for SOME/IP vSomeIP Service Setup builds or starts without missing dependencies.
  • Access to vSomeIP, CMake, Wireshark is confirmed.
  • Required artifacts are present: vSomeIP repo, service config, network route.
  • Setup ticket contains bootstrap output and final verification result.

Typical Setup Problems

  • Outdated bootstrap commands can create inconsistent local environments.
  • Missing service config usually causes setup to pass locally but fail in integration.
  • Access should be removed when the engineer leaves the project or supplier engagement.

When To Ask For Help

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to Middleware Team. The usual requester is the engineer joining the project, and the standard service desk or program channel should hold the final note.
