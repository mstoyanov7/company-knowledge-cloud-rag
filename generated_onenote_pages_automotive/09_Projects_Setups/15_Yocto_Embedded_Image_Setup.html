Yocto Embedded Image Setup

Sets up a Yocto image workspace for embedded Linux target builds and board support package validation.

Project Context

Yocto Embedded Image Setup is used when an engineer needs a working local copy of the project, not just read access to the repository. This page sets up a Yocto image workspace for embedded Linux target builds and board support package validation.

The setup normally starts with access to Yocto, then moves through the required artifacts: Yocto layers, BSP manifest, image recipe. The important part is to keep the machine, the project configuration, and the verification output aligned. If those three do not match, the project can appear to work locally and still fail during integration.

Embedded Platform keeps this page current because small toolchain changes matter in automotive work. A different compiler, DBC file, ODX package, schema version, or bench profile can change the result enough to waste a full validation session.

Before Opening a Terminal

Confirm that your account is already in the right access group for Yocto. Do that before opening a terminal, because clone and package errors often look like network problems when they are actually permission problems.

Create a clean workspace and avoid reusing an old directory from another vehicle program. Keep Yocto layers and BSP manifest next to the setup ticket so another engineer can reproduce the same run later.

Windows Setup

Use this path on a company Windows laptop or before switching into WSL.

$workspace = Join-Path $env:USERPROFILE "automotive\yocto-embedded-image-setup"
New-Item -ItemType Directory -Force $workspace | Out-Null
Set-Location $workspace
git config --global core.longpaths true
wsl --install -d Ubuntu-22.04
wsl bash -lc "sudo apt update && sudo apt install -y git repo gawk wget diffstat unzip texinfo gcc build-essential chrpath socat cpio python3 python3-pip xz-utils zstd"
wsl bash -lc "mkdir -p ~/automotive/yocto-embedded-image-setup && cd ~/automotive/yocto-embedded-image-setup && repo init -u ssh://git.company.local/embedded/yocto-manifest.git -b kirkstone"
wsl bash -lc "cd ~/automotive/yocto-embedded-image-setup && repo sync -c --jobs=8"

Yocto Image Build

Use this path on Linux, WSL, or a standard validation workstation.

mkdir -p ~/automotive/yocto-embedded-image
cd ~/automotive/yocto-embedded-image
repo init -u ssh://git.company.local/embedded/yocto-manifest.git -b kirkstone
repo sync -c --jobs=8
source poky/oe-init-build-env build-auto
bitbake-layers add-layer ../meta-company-automotive
bitbake company-image-minimal

Local Config Example

Keep this file local unless the project owner asks for it in the repository.

MACHINE = "company-qemu-auto"
DISTRO = "company-automotive"
IMAGE_FEATURES += "ssh-server-openssh debug-tweaks"
BB_NUMBER_THREADS = "8"
PARALLEL_MAKE = "-j 8"

Verification Commands

Run verification after setup and again after changing profiles, schemas, or hardware mappings.

source poky/oe-init-build-env build-auto
bitbake company-image-minimal -n
bitbake company-image-minimal

After the First Build

After the first successful build, attach the terminal output and the generated local config to the setup ticket. The expected evidence is setup ticket, access approval, bootstrap output, verification log, plus any project-specific logs created by BitBake or Artifact Registry.

Do not clean the workspace until the verification step has passed once. Keeping the first build directory for a day makes it easier for Embedded Platform to compare cache files, generated code, downloaded packages, or tool versions if something fails for the next engineer.

Checks To Keep With The Ticket

  • Workspace for Yocto Embedded Image Setup builds or starts without missing dependencies.
  • Access to Yocto, BitBake, Artifact Registry is confirmed.
  • Required artifacts are present: Yocto layers, BSP manifest, image recipe.
  • Setup ticket contains bootstrap output and final verification result.

Typical Setup Problems

  • Outdated bootstrap commands can create inconsistent local environments.
  • Missing BSP manifest usually causes setup to pass locally but fail in integration.
  • Access should be removed when the engineer leaves the project or supplier engagement.

When To Ask For Help

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to Embedded Platform. The usual requester is the engineer joining the project, and the standard service desk or program channel should hold the final note.
