Flutter Embedded HMI Setup

Sets up Flutter tooling for embedded HMI screens running on Linux or Android target profiles.

Project Context

Flutter Embedded HMI Setup is used when an engineer needs a working local copy of the project, not just read access to the repository. This page sets up Flutter tooling for embedded HMI screens running on Linux or Android target profiles.

The setup normally starts with access to Flutter SDK, then moves through the required artifacts: Flutter repo, target profile, asset bundle. The important part is to keep the machine, the project configuration, and the verification output aligned. If those three do not match, the project can appear to work locally and still fail during integration.

HMI Platform keeps this page current because small toolchain changes matter in automotive work. A different compiler, DBC file, ODX package, schema version, or bench profile can change the result enough to waste a full validation session.

Before Opening a Terminal

Confirm that your account is already in the right access group for Flutter SDK. Do that before opening a terminal, because clone and package errors often look like network problems when they are actually permission problems.

Create a clean workspace and avoid reusing an old directory from another vehicle program. Keep Flutter repo and target profile next to the setup ticket so another engineer can reproduce the same run later.

Windows Setup

Use this path on a company Windows laptop or before switching into WSL.

$workspace = Join-Path $env:USERPROFILE "automotive\flutter-embedded-hmi-setup"
New-Item -ItemType Directory -Force $workspace | Out-Null
Set-Location $workspace
git config --global core.longpaths true
winget install -e --id Git.Git
winget install -e --id Google.Flutter
winget install -e --id Google.AndroidStudio
git clone ssh://git.company.local/hmi/flutter-embedded-hmi.git .
Write-Output "Run the Linux or project-specific verification block after tools are installed."

Flutter Target Setup

Use this path on Linux, WSL, or a standard validation workstation.

mkdir -p ~/automotive/flutter-embedded-hmi
cd ~/automotive/flutter-embedded-hmi
git clone ssh://git.company.local/hmi/flutter-embedded-hmi.git .
flutter config --enable-linux-desktop
flutter pub get
flutter test
flutter build linux --debug --dart-define=TARGET_PROFILE=bench

Verification Commands

Run verification after setup and again after changing profiles, schemas, or hardware mappings.

flutter doctor
flutter test
flutter analyze

After the First Build

After the first successful build, attach the terminal output and the generated local config to the setup ticket. The expected evidence is setup ticket, access approval, bootstrap output, verification log, plus any project-specific logs created by Android Studio or Artifact Registry.

Do not clean the workspace until the verification step has passed once. Keeping the first build directory for a day makes it easier for HMI Platform to compare cache files, generated code, downloaded packages, or tool versions if something fails for the next engineer.

Checks To Keep With The Ticket

  • Workspace for Flutter Embedded HMI Setup builds or starts without missing dependencies.
  • Access to Flutter SDK, Android Studio, Artifact Registry is confirmed.
  • Required artifacts are present: Flutter repo, target profile, asset bundle.
  • Setup ticket contains bootstrap output and final verification result.

Typical Setup Problems

  • Outdated bootstrap commands can create inconsistent local environments.
  • Missing target profile usually causes setup to pass locally but fail in integration.
  • Access should be removed when the engineer leaves the project or supplier engagement.

When To Ask For Help

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to HMI Platform. The usual requester is the engineer joining the project, and the standard service desk or program channel should hold the final note.
