OEM Supplier Delivery Package Setup

Sets up the supplier delivery package workspace for release notes, evidence, and interface artifacts.

What This Workspace Is For

OEM Supplier Delivery Package Setup is used when an engineer needs a working local copy of the project, not just read access to the repository. This page sets up the supplier delivery package workspace for release notes, evidence, and interface artifacts.

The setup normally starts with access to Supplier Portal, then moves through the required artifacts: delivery package, supplier manifest, evidence folder. The important part is to keep the machine, the project configuration, and the verification output aligned. If those three do not match, the project can appear to work locally and still fail during integration.

Supplier Integration keeps this page current because small toolchain changes matter in automotive work. A different compiler, DBC file, ODX package, schema version, or bench profile can change the result enough to waste a full validation session.

For Windows laptops, use the PowerShell section first and keep long paths enabled in Git. For Linux laptops or WSL, use the shell section. If the project touches lab equipment, check the reservation or bench status before running anything that talks to hardware.

Windows Setup

Use this path on a company Windows laptop or before switching into WSL.

$workspace = Join-Path $env:USERPROFILE "automotive\oem-supplier-delivery-package-setup"
New-Item -ItemType Directory -Force $workspace | Out-Null
Set-Location $workspace
git config --global core.longpaths true
winget install -e --id Git.Git
winget install -e --id Python.Python.3.12
git clone ssh://git.company.local/supplier/delivery-package-template.git .
Write-Output "Run the Linux or project-specific verification block after tools are installed."

Delivery Package

Use this path on Linux, WSL, or a standard validation workstation.

mkdir -p ~/automotive/supplier-delivery-package
cd ~/automotive/supplier-delivery-package
git clone ssh://git.company.local/supplier/delivery-package-template.git .
python tools/create_package.py --release RELEASE-ID --supplier SUPPLIER-ID
python tools/validate_manifest.py delivery/manifest.yaml
zip -r delivery-package.zip delivery evidence release-notes.md

pytest tests -q
tar -czf oem-supplier-delivery-package-setup-setup-evidence.tgz logs reports

How Engineers Usually Work With It

Engineers usually start this workspace during onboarding, release preparation, or a supplier handoff. The first run should be treated like a controlled setup: download the project material, install the exact tools, run the verification command, and then save the output.

Once the project is running, daily work should happen on a feature branch with the same local profile used during setup. If evidence folder changes, repeat the verification command before blaming the application code.

Ready Check

  • Workspace for OEM Supplier Delivery Package Setup builds or starts without missing dependencies.
  • Access to Supplier Portal, Artifact Registry, Requirements Traceability is confirmed.
  • Required artifacts are present: delivery package, supplier manifest, evidence folder.
  • Setup ticket contains bootstrap output and final verification result.

Support Path

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to Supplier Integration. The usual requester is the engineer joining the project, and the standard service desk or program channel should hold the final note.
