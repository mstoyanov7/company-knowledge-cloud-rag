OTA Campaign Simulator Setup

Sets up an OTA campaign simulator for dry-run validation of cohort targeting and update states.

Project Context

OTA Campaign Simulator Setup is used when an engineer needs a working local copy of the project, not just read access to the repository. This page sets up an OTA campaign simulator for dry-run validation of cohort targeting and update states.

The setup normally starts with access to OTA Console, then moves through the required artifacts: campaign simulator, cohort data, state machine config. The important part is to keep the machine, the project configuration, and the verification output aligned. If those three do not match, the project can appear to work locally and still fail during integration.

OTA Operations keeps this page current because small toolchain changes matter in automotive work. A different compiler, DBC file, ODX package, schema version, or bench profile can change the result enough to waste a full validation session.

Before Opening a Terminal

Confirm that your account is already in the right access group for OTA Console. Do that before opening a terminal, because clone and package errors often look like network problems when they are actually permission problems.

Create a clean workspace and avoid reusing an old directory from another vehicle program. Keep campaign simulator and cohort data next to the setup ticket so another engineer can reproduce the same run later.

Windows Setup

Use this path on a company Windows laptop or before switching into WSL.

$workspace = Join-Path $env:USERPROFILE "automotive\ota-campaign-simulator-setup"
New-Item -ItemType Directory -Force $workspace | Out-Null
Set-Location $workspace
git config --global core.longpaths true
winget install -e --id Git.Git
winget install -e --id Python.Python.3.12
git clone ssh://git.company.local/ota/campaign-simulator.git .
Write-Output "Run the Linux or project-specific verification block after tools are installed."

OTA Simulator

Use this path on Linux, WSL, or a standard validation workstation.

mkdir -p ~/automotive/ota-campaign-simulator
cd ~/automotive/ota-campaign-simulator
git clone ssh://git.company.local/ota/campaign-simulator.git .
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python simulator/load_cohort.py data/cohorts/bench-vehicles.csv
python simulator/run_campaign.py configs/dry-run.yaml

After the First Build

After the first successful build, attach the terminal output and the generated local config to the setup ticket. The expected evidence is setup ticket, access approval, bootstrap output, verification log, plus any project-specific logs created by Python or Telemetry Dashboard.

Do not clean the workspace until the verification step has passed once. Keeping the first build directory for a day makes it easier for OTA Operations to compare cache files, generated code, downloaded packages, or tool versions if something fails for the next engineer.

Checks To Keep With The Ticket

  • Workspace for OTA Campaign Simulator Setup builds or starts without missing dependencies.
  • Access to OTA Console, Python, Telemetry Dashboard is confirmed.
  • Required artifacts are present: campaign simulator, cohort data, state machine config.
  • Setup ticket contains bootstrap output and final verification result.

Typical Setup Problems

  • Outdated bootstrap commands can create inconsistent local environments.
  • Missing cohort data usually causes setup to pass locally but fail in integration.
  • Access should be removed when the engineer leaves the project or supplier engagement.

When To Ask For Help

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to OTA Operations. The usual requester is the engineer joining the project, and the standard service desk or program channel should hold the final note.
