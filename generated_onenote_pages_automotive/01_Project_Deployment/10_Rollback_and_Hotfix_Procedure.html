Rollback and Hotfix Procedure

Explains rollback decision criteria, hotfix ownership, and communication steps for failed production releases.

Release Context

Rollback and Hotfix Procedure is written for the moment when engineering work is almost finished but production risk is still real. This page explains rollback decision criteria, hotfix ownership, and communication steps for failed production releases.

The release owner should read it before the deployment window starts, not during the window. The page connects the release material with GitHub Enterprise, Incident Desk, and Status Page so the team can see what is moving and how rollback will work.

How The Release Usually Moves

The normal path starts with rollback plan, then confirms hotfix branch, and finally checks customer impact note. When one of those is missing, the release should stay in preparation rather than becoming a production incident.

During the window, Release Management keeps the handoff short: what changed, what dashboard is watched, who can stop the rollout, and where the final note will be attached.

Gate Checks

  • Release scope, deployment window, and responsible owner are recorded.
  • Deployment evidence includes rollback plan, hotfix branch, customer impact note.
  • Rollback owner and monitoring path are known before rollout.
  • Post-deployment checks are complete before handoff.

Evidence Trail

Keep release approval, validation evidence, monitoring plan, rollback owner together with the release ticket. The release owner should be able to open one record and see what was deployed, who approved it, which monitoring view was used, and who owns rollback.

Release Risks

  • Deployment without rollback ownership increases recovery time.
  • Release evidence must be current for the exact software package being deployed.
  • Monitoring gaps should block rollout until ownership is clear.

Escalation

Escalate unclear ownership, missing evidence, blocked approvals, or exceptions to Release Management. The usual requester is the release owner or program lead, and the standard service desk or program channel should hold the final note.
